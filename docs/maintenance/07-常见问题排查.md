# 常见问题排查

## 快速定位顺序

1. 先看用户看到的页面和具体报错。
2. 再看对应 API 是否返回错误码。
3. 再看 store 里任务、订单、图片、积分状态。
4. 再看 worker/provider/storage 日志。
5. 最后才改 UI 文案。

不要一上来改前端。很多问题根因在 API、worker、provider 或配置。

## 常见症状

| 症状 | 优先检查 |
| --- | --- |
| 生成一直转圈 | API 任务状态、worker 是否 claim、provider 是否超时 |
| 生成成功但页面没图 | taskId 恢复、images 返回、thumbnailUrl、前端 snapshot |
| 再次生成显示旧图 | generate 页 task/images 清理和 submitting guard |
| 切页回来占位数量乱闪 | active task snapshot 和 task.quantity |
| 邮件收不到 | `MAILER_PROVIDER`、SMTP 配置、API 日志 |
| 支付后没到账 | payment event、order 状态、credit ledger |
| 历史页没有图片 | `/api/images`、软删除、用户权限、thumbnail |
| 后台数据不准 | admin API 聚合逻辑和 store 数据 |

## 常用验证命令

窄范围优先：

```powershell
node --test --test-name-pattern "generate page" tests\web-routes.test.mjs
node --test tests\maintenance-map.test.mjs
```

类型和构建：

```powershell
npm run typecheck
npm run build
```

全量测试：

```powershell
npm test
```

如果全量测试失败，先判断是不是当前改动相关。不要把无关旧失败混进本次交付结论。

## 文档同步

改动以下内容后，必须同步维护文档或自动地图：

- API 路由。
- 页面路由。
- 生成任务状态机。
- 支付/积分流程。
- 邮箱验证和登录流程。
- provider 或 storage 契约。
- 新增环境变量。

自动地图更新：

```powershell
npm run docs:maintenance
```
