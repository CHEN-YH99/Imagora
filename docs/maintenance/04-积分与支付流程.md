# 积分与支付流程

## 积分账户

核心数据：

- `CreditAccount`：用户余额、总获取、总消耗。
- `CreditLedgerEntry`：每一次发放、扣减、退回、过期和人工调整。

图片生成扣积分时，不只改余额，还必须写流水。维护时不能只看 `balance`。

## 图片生成扣减

生成任务创建时会：

1. 根据参数 quote 成本。
2. 检查余额是否足够。
3. 扣减积分。
4. 写 generation task。
5. 写 credit ledger。

如果任务最终没有交付图片，补偿逻辑会尝试退回积分。

## 套餐和订单

套餐和订单主要入口：

- `/pricing`
- `/orders`
- `POST /api/orders`
- `POST /api/orders/:orderId/pay`

支付 provider 由 `packages/payments` 适配。当前有 mock 和 Stripe 等 provider 结构。

## Webhook 和对账

真实支付不能只依赖前端回跳。维护时要同时看：

- 支付创建接口。
- webhook 验签。
- payment event 记录。
- order 状态更新。
- credit account 入账。
- 对账脚本和管理后台维护入口。

## 常见维护坑

- 订单已支付但余额未到账：先查 payment event，再查 order，再查 ledger。
- 重复回调：必须保持幂等，不要重复发积分。
- 金额不一致：不要自动发积分，必须走异常和人工处理。
- 支付 provider 切换：同步检查生产配置校验和 release drill。
