# 图片生成主链路

## 主流程

入口页面：`apps/web/app/generate/page.tsx`

核心链路：

```text
/generate 表单
-> POST /api/generation/quote
-> POST /api/generation/tasks
-> queue 提交任务
-> apps/worker/src/main.ts claim 任务
-> packages/ai-providers 调用图片模型
-> packages/storage 保存图片和缩略图
-> API 轮询任务状态
-> /history、/images/[imageId] 展示结果
```

## 前端状态边界

生成页同时管理几类状态：

- 表单草稿：prompt、negativePrompt、aspectRatio、quantity、quality、model
- 当前任务：task、images
- 提交中状态：loading、submittingGenerationRef
- 恢复状态：taskId URL、active task id、task snapshot
- 预览状态：selectedPreviewImage

这几类状态不能混用。尤其是 `taskId` 恢复和新提交之间有竞态，改动时必须验证：

- 再次生成时旧成功图片不能盖住新 loading。
- 改提示词后 POST body 必须是新提示词。
- 切页再回生成页时，占位卡片数量必须等于当前任务数量。
- active task snapshot 不能被旧 URL 恢复逻辑清掉。

## API 创建任务

接口：`POST /api/generation/tasks`

职责：

- 校验登录。
- 按配置校验邮箱是否验证。
- 解析模型和生成参数。
- 计算积分成本。
- 创建 generation task。
- 扣减积分并写积分流水。
- 入队或 inline 触发 worker。

这一步只负责创建任务，不负责长时间等待 provider 完成。

## Worker 执行

worker 的稳妥模式是两段式：

1. 短事务 claim：`PENDING -> RUNNING`
2. 锁外执行 provider 和 storage
3. 短事务 finalize：写结果或失败

不要让长时间 provider 调用占住 store 写锁，否则 GET 轮询会卡住。

## Provider 约束

OpenAI-compatible 图片 provider 需要明确请求和响应契约。维护时重点看：

- 请求是否显式要求图片数据格式。
- 响应 parser 是否兼容实际 gateway 返回。
- 失败时是否写入可读 failure code/message。

provider 问题不要先改 UI 文案，先确认出站请求和入站 payload。

## 恢复与轮询

生成页支持：

- `taskId` URL 恢复。
- `sessionStorage` 保存 task snapshot。
- active task id 用于切页回来继续展示生成中任务。

这块是高风险区域。改动后至少跑：

```powershell
node --test --test-name-pattern "generate page" tests\web-routes.test.mjs
```

涉及 worker 或 API 锁时，还要跑：

```powershell
node --test tests\api-flow.test.mjs
```
