# 认证与邮箱验证流程

## 注册流程

入口：`apps/web/app/register/page.tsx`

后端：`POST /api/auth/register`

注册时 API 会：

1. 创建用户，`emailVerifiedAt` 初始为 `null`。
2. 创建登录 session，并写入 HttpOnly cookie。
3. 创建邮箱验证 token。
4. 只保存 token hash，不保存明文 token。
5. 拼出 `/verify-email?token=...` 链接。
6. 调用 `packages/mailer` 发送验证邮件。
7. 返回 `emailDelivered` 给前端，用于区分发信成功和发信失败提示。

注册成功后用户已经登录，但邮箱未验证。

## 验证链接流程

入口页面：`apps/web/app/verify-email/page.tsx`

接口：`POST /api/auth/verify-email`

页面读取 URL 上的 `token`，提交给 API。API 会把 token 再 hash，查找未使用且未过期的记录。验证通过后：

- 写入 `user.emailVerifiedAt`
- 写入 token 的 `usedAt`
- 返回验证成功

token 默认有效期由 `EMAIL_VERIFICATION_TOKEN_TTL_HOURS` 控制，未配置时按 24 小时。

## 重新发送验证邮件

入口页面：`apps/web/app/account/page.tsx`

接口：`POST /api/auth/resend-verification`

账号页发现 `emailVerifiedAt` 为空时展示“重新发送验证邮件”。重发时 API 会清掉该用户旧的未使用 token，再生成新 token 和新链接。

## 生成前邮箱验证门槛

图片生成创建接口 `POST /api/generation/tasks` 会调用邮箱验证检查。是否强制检查由 `REQUIRE_EMAIL_VERIFICATION` 控制：

- 开发/测试默认不强制。
- 生产默认强制。

未验证用户在生产环境提交生成会返回 `EMAIL_NOT_VERIFIED`。

## 邮件 provider

`packages/mailer/src/index.ts` 根据 `MAILER_PROVIDER` 创建 mailer：

- `console`：默认，只打印日志，不真实投递。
- `smtp`：真实 SMTP 投递。

真实投递至少需要：

- `MAILER_PROVIDER=smtp`
- `SMTP_HOST`
- `SMTP_PORT`
- `SMTP_USER`
- `SMTP_PASSWORD`
- `SMTP_FROM`

维护时不要只看“代码有发信函数”就判断邮件正常，必须看运行时 provider 和日志。
