# Imagora 维护总览

这组文档给后续维护者看，目标是快速理解代码边界、主流程和排查入口。不要把这里写成产品介绍；这里讲的是代码如何协作、哪里容易坏、改完怎么验。

## 项目定位

Imagora 是图片生成 SaaS。当前主链路包括账号认证、邮箱验证、图片生成任务、worker 处理、图片存储、历史/收藏/下载、积分支付、管理后台和安全审核。

## 代码分层

| 区域 | 责任 |
| --- | --- |
| `apps/web` | Next.js 前端页面、API client、用户交互状态 |
| `apps/api` | Fastify API、认证、任务创建、支付、后台接口 |
| `apps/worker` | 生成任务执行、provider 调用、图片落库 |
| `packages/database` | store 抽象、Prisma schema、JSON store 兼容 |
| `packages/ai-providers` | OpenAI-compatible 图片 provider 适配 |
| `packages/storage` | 本地/S3/R2 文件存储与签名 URL |
| `packages/payments` | Stripe/mock 支付 provider |
| `packages/mailer` | Console/SMTP 邮件发送 |
| `packages/safety` | 安全规则与内容审核 |
| `packages/queue` | inline/队列任务抽象 |

## 自动生成地图

`docs/maintenance/generated` 目录由 `npm run docs:maintenance` 生成，包含：

- `api-routes.md`：`apps/api/src/main.ts` 里的 API 路由列表
- `web-pages.md`：`apps/web/app/**/page.tsx` 页面路由列表
- `package-map.md`：`packages/*` workspace 包列表
- `env-vars.md`：源码中的环境变量引用位置

这些文件不要手改。改代码后重新运行命令生成。

## 维护原则

- 先定位边界，再改代码。前端状态、API store、worker、provider、storage 是不同层。
- 改图片生成链路时，必须同时考虑创建任务、轮询、刷新恢复、worker 执行和历史展示。
- 改支付/积分时，必须同时看订单状态、积分流水、幂等和对账脚本。
- 改邮箱或登录时，必须同时看 cookie、token、前端提示和生产配置。
- 文档只记录长期有用的流程和坑，不写“设置变量”这类废话注释。
