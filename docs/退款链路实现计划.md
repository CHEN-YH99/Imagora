# 任务 4 · 退款链路实现计划

> 依据：2026-07-15 六维审计 P0#4「退款链路完全缺失」。前端已渲染 REFUNDED 文案/badge/筛选，后端整条链路（退款 API、Stripe refund、积分回收、状态写入）为空。
> 决策已定：① 积分回收**允许扣成负数**（余额不足也退，用户下次充值先抵欠账）；② 退款信息**不加 schema 字段**，靠现有四条链路承载（零迁移）。

---

## 一、设计原则（复用现有范式，不造新轮子）

退款是 `applyPaymentSucceeded` 的镜像操作，完全对齐它的成熟范式：

| 维度 | 支付（已有 `applyPaymentSucceeded`） | 退款（本次新增 `applyPaymentRefunded`） |
|---|---|---|
| 状态流转 | `PENDING/CLOSED → PAID` | `PAID → REFUNDED` |
| 幂等键 | `order-grant:{orderId}` | `order-refund:{orderId}` |
| 事件记录 | paymentEvents 记 `payment.succeeded` | paymentEvents 记 `payment.refunded` |
| 积分动作 | `grantCredits(+plan.credits)` | `grantCredits(−已发积分)` ← 传负数 |
| 事件去重 | `provider+providerEventId` | 同上 |

**关键复用**：`grantCredits`（main.ts:1212）只做 `account.balance += amount`，不校验余额，传负数即回收，天然支持「扣成负数」策略。`adjustCredits` 反而会拦负余额（1262），不能用。

**追溯性**（不加字段也够）：
- 退款事件 → `paymentEvents`（含 refundReason、refundedAmountCents 在 payload）
- 积分回收 → `creditLedgerEntries`（一条负数 GRANT）
- 操作人/原因 → admin `audit()` 日志
- 退款时间 → `order.updatedAt`（status===REFUNDED 时即退款时刻）

## 二、四层改动清单

### 第 1 层 · payments 包（`packages/payments/src/index.ts`）

**新增接口类型**：
```
RefundOrderInput  { orderId, orderNo, paymentIntentId, amountCents, currency, reason? }
RefundResult      { status: "refunded" | "failed", providerRefundId: string | null, detail?: string }
```

**`PaymentProvider` 接口加方法**：`refundOrder(input: RefundOrderInput): Promise<RefundResult>`
- 约定：失败返回 `status:"failed"` + detail，**不抛错**（对齐 `retrieveOrderPaymentStatus` 的约定，退款在事务外调，抛错会中断）

**三个实现**：
- **Stripe**：调 `POST /v1/refunds`，body `payment_intent={id}`（全额退，v1 不做部分退）。复用现有 `apiBaseUrl`/`secretKey`/`timeoutMs`/超时与错误处理范式。留 `// TODO(verify-with-live-stripe)` 锚点。
  - ⚠️ 注意：Stripe 下 `paymentIntentId` 存的其实是 **checkout session id**（见 createPayment）。退款需真实 payment_intent —— 需先 `GET /v1/checkout/sessions/{id}` 取 `payment_intent`，再对其发 refund。这一步会在实现里处理。
- **Mock**：走 env `MOCK_REFUND_FAIL_ORDERS` 控制可测失败路径，否则返回 `refunded` + `mock_refund_{orderId}`。
- **Wechat/Alipay**：占位补桩（`status:"failed"` + not implemented），维持接口完整、不被 createPaymentProvider 返回。

### 第 2 层 · 退款核心函数（`apps/api/src/main.ts`）

新增 `applyPaymentRefunded(data, input)`，镜像 `applyPaymentSucceeded`：

1. **幂等去重**：`provider+providerEventId`（refund 事件）已存在 → 返回 `DUPLICATE_EVENT`
2. **订单校验**：找单；`order.status !== "PAID"` → 返回 `ORDER_NOT_REFUNDABLE`（只有已付款单可退）
3. **写 paymentEvents**：记一条 `payment.refunded`，payload 含 refundReason / refundedAmountCents / providerRefundId
4. **回收积分**：`grantCredits(data, userId, −refundedCredits, "ORDER", orderId, "order-refund:{orderId}", "Refund: {reason}")`
   - `refundedCredits` = 当初这单发的积分（从 plan.credits 或 order-grant 账本条目读，取实际发放值，防 plan 改动后金额不符）
5. **状态写入**：`order.status = "REFUNDED"`，`order.updatedAt = now`
6. **返回**：`{ order, balanceAfter, refunded, duplicateEvent, reason }`（结构对齐 applyPaymentSucceeded）

> 积分回收金额取「当初实际发放值」而非「当前 plan.credits」——查 `order-grant:{orderId}` 那条账本 entry 的 amount，plan 后来改了也不影响。

### 第 3 层 · 退款 API 路由（`apps/api/src/routes/admin.ts`）

新增 `POST /api/admin/orders/:orderId/refund`（对齐现有 admin 范式）：
- `requireAdmin` 鉴权
- schema：`{ reason: string(必填, 用于审计), providerRefundId? }`
- 流程：
  1. **事务外**先调 `paymentProvider.refundOrder(...)` 真实退款（网络 IO 不进事务）
  2. refund 成功 → `store.update` 里调 `applyPaymentRefunded` 写状态+回收积分
  3. refund 失败 → 返回 4xx/5xx，记 `recordOperationalIncident`，**不改订单状态**（钱没退成绝不置 REFUNDED）
  4. `audit(admin.id, "order.refund", ...)` 记操作人/原因/金额
- 返回 `envelope(request, { order, balanceAfter, refunded })`

**新增 schema**（`apps/api/src/schemas.ts`）：`refundOrderSchema = z.object({ reason: z.string().trim().min(1).max(200), providerRefundId: z.string().optional() })`

### 第 4 层 · 前端 admin 入口（`apps/web`）

- `lib/api.ts`：加 `refundOrder(orderId, reason)` 封装
- `app/admin/page.tsx`：订单详情/列表加「退款」按钮（仅对 PAID 单显示），点击弹确认框输入原因 → 调 refundOrder → 刷新
  - 复用页面现有的确认框/错误处理范式

> 前端已有的 REFUNDED 文案/badge/筛选（orders/page.tsx、admin/page.tsx）此时就真能对上数据了——不用改，接上后端即生效。

## 三、边界情况处理

| 场景 | 处理 |
|---|---|
| 重复点退款 | `order-refund:{orderId}` 幂等键 + refund 事件去重，第二次返回 DUPLICATE，不重复退 |
| 退未付款单（PENDING/CLOSED） | `ORDER_NOT_REFUNDABLE`，拒绝 |
| 用户积分已花光 | `grantCredits` 传负数照扣，**余额可为负**（既定策略），账本留负数 GRANT 条目 |
| Stripe refund 网络失败 | refundOrder 返回 failed → 路由返回错误 + 记 incident，**订单状态不变**（不会出现「钱没退成但标了 REFUNDED」） |
| plan 被删/改 | 积分回收取 order-grant 账本实际发放值，不依赖当前 plan.credits |
| Stripe paymentIntentId 实为 session id | 退款前先反查 session 取真实 payment_intent，再退 |

## 五、不做什么（划清边界）

- ❌ 不做部分退款（v1 只全额退，Stripe refund 不传 amount）
- ❌ 不加 Order schema 字段、不写迁移（既定决策）
- ❌ 不做用户自助退款（仅管理员发起，用户侧退款申请是另一个需求）
- ❌ 不碰 PrismaStore 整库读写（那是第 2 档独立大任务）
