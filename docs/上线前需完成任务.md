  - 微信/支付宝 provider 仍是预留桩，启用这些支付方式前必须补对应退款实现；当前生产支付 provider 目标是 Stripe

### ✅ P1-2. generation enqueue 可靠补偿
- **病根**：任务与扣费提交后才在事务外调用 Redis enqueue；API 若在两者之间崩溃，任务会永久停在 `PENDING`。原先 enqueue 失败还会立即把任务标记失败并退款，客户端用同一 `clientRequestId` 重试时容易得到失败任务，无法可靠恢复。
- **完成内容**：
  - 生成任务和扣费先持久化；即时 enqueue 失败不回滚、不退款，接口仍返回已创建的 `PENDING` 任务
  - API 启动时和后台定时扫描 `PENDING` 任务补投；BullMQ 使用 `jobId=taskId` 保证重复补投幂等
  - producer 增加 Redis 命令超时，失败客户端主动丢弃并在下一次请求重建，避免断线客户端长期卡死
  - `clientRequestId` 重试复用原任务，不重复创建任务、扣费或退款
  - 补偿扫描按 `(createdAt, id)` 轮转游标分批，批次后的任务不会因前一批长期 `PENDING` 而饿死；环绕后仍会继续重试旧失败任务
  - shutdown 先停止新扫描，并等待活动补偿和所有即时 enqueue 完成，再关闭 BullMQ 客户端；停止后的调用不再访问 store/queue
- **Review 修复**：
  - 修复固定读取最老批次导致后续 `PENDING` 永久饥饿的问题
  - 修复 shutdown 只等待后台补偿、不等待请求触发即时 enqueue 的关闭竞态

  - [apps/web/app/history/page.tsx](../apps/web/app/history/page.tsx) 为 `copyPrompt()` 和 `toggleFavorite()` 补 `try/catch`
  - 收藏状态以 API 返回的 `result.favorite` 为准，不再本地盲目取反
- **Review 结论**：
  - API 失败时不会把 UI 乐观状态误提交为事实，收藏状态以服务端结果收敛
  - 剪贴板不可用时页面不会抛出未处理异常
- **验证**：✅ `tests/web-routes.test.mjs` 覆盖异常处理静态路径；✅ 本轮全量门禁通过

### ✅ P2-4. verify-email 增加限流
- **病根**：`POST /api/auth/verify-email` 未纳入运行时限流，攻击者可以高频撞 token 或刷验证请求。
- **完成内容**：[apps/api/src/rate-limit-runtime.ts](../apps/api/src/rate-limit-runtime.ts) 新增 `auth-verify-email` 规则，覆盖 `POST /api/auth/verify-email`。

  - 与现有认证类接口限流模式一致，失败语义保持统一
- **验证**：✅ `tests/api-route-modules.test.mjs` 覆盖 verify-email 限流匹配；✅ 本轮全量门禁通过

### ✅ P2-5. password reset 新令牌作废旧令牌
- **病根**：同一用户多次申请密码重置时，旧的未使用 reset token 不会失效；邮箱泄漏或旧链接被转发时，风险窗口被拉长。
- **完成内容**：
  - [apps/api/src/routes/auth.ts](../apps/api/src/routes/auth.ts) 在同一个 `store.update()` 内查找用户、作废该用户全部未使用旧 token，并生成新 token
  - 邮箱不存在时仍返回成功响应，保持防枚举语义
- **Review 结论**：
  - 只作废同一用户未使用旧 token，不影响其他用户和已使用记录
  - token 作废与新 token 写入在同一 store 更新闭包内完成，避免中间状态泄漏
- **验证**：✅ `tests/auth-flow.test.mjs` 覆盖“新 reset token 作废旧未使用 token”；✅ 本轮全量门禁通过

### ✅ P2-6. admin 列表接口先鉴权再解析 query
- **病根**：部分 admin GET 列表接口在鉴权前先解析 query；虽然多数只是校验失败，但安全边界应该先判断身份，再处理用户输入。
- **完成内容**：
  - [apps/api/src/routes/admin.ts](../apps/api/src/routes/admin.ts) 将 `/api/admin/users`、`/generation/tasks`、`/images`、`/orders`、`/audit-logs`、`/safety-events`、`/safety-appeals` 调整为先 `requireAdmin(request)`，再 parse query
- **Review 结论**：
  - 未授权请求不会触发 admin query parse 路径，接口安全顺序更清晰
  - 已授权请求的查询参数行为保持不变
- **验证**：✅ `tests/api-route-modules.test.mjs` 覆盖 admin 列表接口鉴权先于 query parse；✅ 本轮全量门禁通过

### ✅ P2-7. payment webhook 增加限流

  - 默认阈值独立成环境变量，后续可按真实支付平台重试频率调参
- **验证**：✅ `tests/api-route-modules.test.mjs` 覆盖 webhook 限流匹配；✅ 本轮全量门禁通过
