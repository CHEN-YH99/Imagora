## 第 0 档 · 开机级（照文档部署必崩，最先清）✅ 已完成

### ✅ 1. 部署脚本不加载 `.env.production`
- **病根**：compose 全是 `${VAR:?}` 强制替换，但部署脚本没喂 `--env-file`，docker 默认只读 `.env` 不读 `.env.production` → `up` 都进不去。
- **出处**：运维审计
- **完成内容**：
  - [infra/scripts/deploy-local-prod.mjs](../infra/scripts/deploy-local-prod.mjs) 加 `--env-file .env.production`
  - 加文件存在性检查，缺文件中文报错提示 `cp .env.production.example`
  - 支持 `DEPLOY_ENV_FILE=path` 覆盖
- **验证**：🧪 `docker compose config` 校验通过；未端到端起容器（本机 Docker daemon 未启动）

### ✅ 2. 告警变量没透传进容器 → API 启动即自杀
- **病根**：[apps/api/src/production-config.ts:64](../apps/api/src/production-config.ts) 生产强制要求 `ALERT_WEBHOOK_URL` 或 `ALERT_EMAIL_TO` 至少一个，否则抛错退出；但 compose api 的 environment 块没透传这些变量。
- **出处**：运维审计
- **完成内容**：
  - [infra/docker-compose.prod.yml](../infra/docker-compose.prod.yml) api environment 块补 `ALERT_WEBHOOK_URL`/`ALERT_EMAIL_TO`/`ALERT_COOLDOWN_MS`/`ALERT_EVALUATION_INTERVAL_MS`
  - 前两个用可选透传（不加 `:?`），因为只要求二选一，强制必填会误伤
  - email 通道复用 api 已有的 SMTP 配置
- **验证**：🧪 同上，config 校验通过

### ✅ 3. 空库无迁移编排 → 首部署表不存在崩
- **病根**：`prisma migrate deploy` 脚本存在（[packages/database/package.json:20](../packages/database/package.json)）但全链路无人调用，Dockerfile CMD 直接起 `node main.js`。
- **出处**：运维审计 + 数据层审计
- **完成内容**：
  - [infra/docker-compose.prod.yml](../infra/docker-compose.prod.yml) 新增独立 `migrate` 一次性 service：复用 Dockerfile.api 镜像跑 `prisma:migrate-deploy`，依赖 postgres 健康，`restart: "no"`
  - api / worker 均加 `migrate: condition: service_completed_successfully` 依赖 → 建表在前
- **验证**：🧪 config 校验通过；runtime 镜像确认带 prisma CLI + migrations 目录；未实跑迁移

### ✅ 1b.（连带缺口）模板缺 POSTGRES 三分量变量
- **病根**：compose 自带 postgres 服务、内部用 `POSTGRES_USER/PASSWORD/DB` 拼 `DATABASE_URL`（指向服务名 `postgres:5432`），但 `.env.production.example` 只有一条指向外部 `postgres.example.com` 的 `DATABASE_URL` → `up` 第一步中断。校验第 0 档时揪出。
- **完成内容**：[.env.production.example](../.env.production.example) 补 `POSTGRES_USER`/`POSTGRES_PASSWORD`/`POSTGRES_DB`

> **第 0 档遗留验证项**：待本机/服务器 Docker daemon 可用后，实跑一次 `npm run deploy:local-prod`，确认迁移建表 + api/worker 起容器全链路通。

## 第 1 档 · 资损与反滥用（真金白银 / 被薅）⬜

### ⬜ 4. 退款链路完全缺失 ⚠️
- **现状**：无退款 API、无 Stripe refund、无订单积分回收，但前端已画好退款文案/状态/筛选。
- **出处**：支付 + 前端审计
- **落点**：`packages/payments/src/index.ts`（加 refund）、`apps/api/src/routes/orders.ts`/`admin.ts`（退款路由）、订单积分回收逻辑、`REFUNDED` 状态写入
- **闭环**：⚠️ 逻辑可纯代码写，但真实 Stripe refund 调用需你的 key 才能验证
- **量**：大

### ⬜ 5. webhook 丢失永久掉单
- **现状**：主发放靠 webhook，无主动向 Stripe 拉状态对账；付款成功但 webhook 丢失 → 用户永远拿不到积分且无告警。
- **出处**：支付审计
- **落点**：payments 层加 `retrieveSession`，对账逻辑主动拉 Stripe 状态
- **闭环**：✅ 纯代码　**量**：中

### ⬜ 6. 订单维护无后台定时器
- **现状**：`runOrderMaintenance`（[apps/api/src/main.ts:649](../apps/api/src/main.ts)）只被 HTTP 路由被动触发，没人访问就不对账/不关过期单/不补积分。
- **出处**：支付审计
- **落点**：挂进后台 `setInterval`，与 `startBackgroundGenerationMaintenance` 对齐
- **闭环**：✅ 纯代码　**量**：小

### ⬜ 7. 默认 `inline` 存储会灌库
- **现状**：`STORAGE_PROVIDER` 默认 inline（[packages/storage/src/index.ts:338](../packages/storage/src/index.ts)），prisma + inline 组合会把 base64 图塞进 Postgres（P2024 老坑）；`validateProductionConfig` 未拦此组合。
- **出处**：数据层审计
- **落点**：production-config 加校验，prisma + inline 直接拒启
- **闭环**：✅ 纯代码　**量**：小

### ⬜ 8. captcha 可 100% 自动求解 🔧
- **现状**：答案写在 SVG 的 `aria-label` + 可见 `text` 里（[apps/api/src/captcha-runtime.ts:96-146](../apps/api/src/captcha-runtime.ts)），爬虫解析即破；配合送 120 积分 + 撞库，门槛几乎归零。
- **出处**：安全审计
- **闭环**：🔧 挪答案出 DOM 能提高门槛，根治需上真图形/行为验证码或三方（hCaptcha/腾讯验证码）——**需你定方案**
- **量**：中

- **现状**：每次读全量加载 20 表、每次写全库 `deleteMany` + `createMany` 重建、外层事务套内层事务再叠全局 advisory 锁（[packages/database/src/index.ts:153-197,448-809](../packages/database/src/index.ts)）。数据一涨 O(N²) 写放大 + 连接池死锁，付费流量必炸。

- **量**：特大　**已确认单独排期，暂不在本轮**

## 第 3 档 · 功能空转与内容合规 ⬜

### ⬜ 10. 申诉管理员入口断链
- **现状**：后端全链路 + 用户侧全齐，但 [apps/web/app/admin/page.tsx](../apps/web/app/admin/page.tsx) grep `appeal`/`申诉` 零命中 → 用户申诉永卡 PENDING，管理员看不到。
- **出处**：前端审计
- **落点**：admin 页接 `GET /api/admin/safety-appeals` + `PATCH /api/admin/safety-appeals/:id`；`lib/api.ts` 补 admin 封装
- **闭环**：✅ 纯代码　**量**：小（最好做）

### ⬜ 11. PENDING 超时误杀
- **现状**：`DEFAULT_PENDING_TASK_TIMEOUT_MS = 5min`（[packages/shared/src/index.ts:412](../packages/shared/src/index.ts)），短于单任务最长 25min 处理时长；队列一积压，合法排队任务被误杀退款、永不执行。
- **出处**：AI 链路审计
- **落点**：调 PENDING 超时与 RUNNING 超时口径对齐
- **闭环**：✅ 纯代码　**量**：小

### ⬜ 12. 图生图伪实现 ⚠️
- **现状**：参考图 URL 当文本拼进 prompt（[packages/ai-providers/src/index.ts:1054-1063](../packages/ai-providers/src/index.ts)），发的是文生图 `/images/generations`，根本没传图。
- **出处**：AI 链路审计
- **闭环**：⚠️ 需先确认上游 `ai.xmiaom.com` 是否支持 `/images/edits` 及参数格式，否则盲写白写——**需你/联网验证上游能力**
- **量**：中

### ⬜ 13. 内容安全有形无实 ⚠️
- **现状**：`LocalSafetyProvider.checkImage` 只看 mimeType 就 PASSED；生产强制 http 但真本事押在仓库里没实现的外部端点，且端点挂了 fail-open 会把所有图打进人工队列 = 可用性 DoS。
- **出处**：安全审计
- **闭环**：⚠️ http 审核服务是运维交付项；代码这头只能调 fail-open 策略——**需外部审核服务**
- **量**：中

### ⬜ 14. worker 无 restart / healthcheck
- **现状**：prod compose 五服务零 `restart`，worker 无 healthcheck；BullMQ 唯一消费者崩了不自愈 → 任务全线超时退款。
- **出处**：运维审计
- **落点**：compose worker 加 `restart` + healthcheck（可选给其余服务补 restart）
- **闭环**：✅ 纯代码　**量**：小

### ⬜ 15. Postgres 零备份 ⚠️
- **现状**：只有 JSON store 备份脚本，生产用 PG 无任何 `pg_dump`/`pg_restore`。
- **出处**：运维审计
- **闭环**：⚠️ 脚本可写，但调度周期/存储位置/保留策略需你定
- **量**：中

## P1 档 · 上线后尽快（埋雷不炸）⬜

- ⬜ worker 无优雅退出 → 每次部署产僵尸 RUNNING 任务卡 30min（AI 链路）
- ⬜ enqueue 在事务外 → 崩溃窗口任务永不入队（AI 链路）
- ⬜ 内存态（captcha/登录令牌/metrics）未迁 Redis → **多副本才炸，单副本不阻塞**（安全 + 运维）
- ⬜ CSRF：Origin 缺失时放行写请求 + SameSite 未强校验（安全）
- ⬜ filesystem 签名密钥弱默认值（安全）
- ⬜ 收藏页拉全部再过滤（老收藏丢）、历史页硬上限 50 无分页（前端）
- ⬜ 下单 `clientRequestId` 可选，去重可绕（支付）
- ⬜ claim 每次跑全量维护 O(N²)（AI 链路）
