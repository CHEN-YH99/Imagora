### ✅ P1-1. worker 优雅退出
- **病根**：worker 收到部署停止信号时原先直接退出，已领取任务会停在 `RUNNING`，直到 30 分钟维护超时才失败退款；BullMQ job 也可能被判 stalled 后重跑。
- **完成内容**：
  - [apps/worker/src/main.ts](../apps/worker/src/main.ts) 监听 `SIGINT`/`SIGTERM`；inline 模式先清轮询定时器再等待当前 tick，BullMQ 模式调用 `worker.close()` 停止领取并等待活动 job 完成
  - 新增 [apps/worker/src/shutdown-runtime.ts](../apps/worker/src/shutdown-runtime.ts)，将退出编排收口为幂等控制器；重复信号不会重复关闭资源或中断首次排空
  - [infra/scripts/worker-supervisor.mjs](../infra/scripts/worker-supervisor.mjs) 向子 worker 转发原始信号，等待 `WORKER_SUPERVISOR_SHUTDOWN_TIMEOUT_MS` 后才强制终止；正常排空退出码为 0，超时强退为 1
  - 生产 worker `stop_grace_period` 为 32 分钟，略大于 supervisor 默认 31 分钟上限，容器运行时不会先于 supervisor 强杀
- **Review 结论**：
  - 当前 BullMQ 版本 `Worker.close(false)` 会先设置 closing、停止继续领取，再等待当前 job 完成
  - worker、supervisor、Compose 三层超时顺序正确；inline/BullMQ 两种运行模式均无继续接单或重复 shutdown 路径
- **验证**：✅ worker typecheck/build；✅ `tests/release-drill.test.mjs`（13/13），包含“重复信号只关闭一次、活动任务未完成前 shutdown 不返回、完成后正常退出”的行为测试；✅ 定向 ESLint / Prettier；✅ `git diff --check`

### ✅ P1-2. generation enqueue 可靠补偿
- **病根**：任务与扣费提交后才在事务外调用 Redis enqueue；API 若在两者之间崩溃，任务会永久停在 `PENDING`。原先 enqueue 失败还会立即把任务标记失败并退款，客户端用同一 `clientRequestId` 重试时容易得到失败任务，无法可靠恢复。
- **完成内容**：
  - 生成任务和扣费先持久化；即时 enqueue 失败不回滚、不退款，接口仍返回已创建的 `PENDING` 任务
  - API 启动时和后台定时扫描 `PENDING` 任务补投；BullMQ 使用 `jobId=taskId` 保证重复补投幂等
  - producer 增加 Redis 命令超时，失败客户端主动丢弃并在下一次请求重建，避免断线客户端长期卡死
  - `clientRequestId` 重试复用原任务，不重复创建任务、扣费或退款
  - 补偿扫描按 `(createdAt, id)` 轮转游标分批，批次后的任务不会因前一批长期 `PENDING` 而饿死；环绕后仍会继续重试旧失败任务
  - shutdown 先停止新扫描，并等待活动补偿和所有即时 enqueue 完成，再关闭 BullMQ 客户端；停止后的调用不再访问 store/queue
- **Review 修复**：
  - 修复固定读取最老批次导致后续 `PENDING` 永久饥饿的问题
  - 修复 shutdown 只等待后台补偿、不等待请求触发即时 enqueue 的关闭竞态
  - 补两个旧客户端请求并发失败的竞态测试，确认旧请求不会误关闭已重建的新 BullMQ 客户端
- **验证**：✅ `build:packages`、API typecheck/build；✅ `tests/performance-config.test.mjs`（10/10）；✅ `tests/api-flow.test.mjs`（12/12），覆盖 Redis 不可用时任务/扣费持久化与重复请求幂等；✅ 定向 ESLint / Prettier；✅ `git diff --check`

### ✅ P1-3. captcha / 登录令牌 / HTTP metrics 迁移 Redis
- **病根**：验证码 challenge、验证码 verification、登录尝试令牌和 HTTP metrics 原先均保存在 API 进程内存；多副本负载均衡时，请求落到不同实例会随机失效，实例重启还会直接丢状态。
- **完成内容**：
  - 新增 [apps/api/src/runtime-state.ts](../apps/api/src/runtime-state.ts)，统一提供 memory/Redis 两种运行时状态实现；生产默认并强制使用 `RUNTIME_STATE_PROVIDER=redis`
  - captcha challenge、verification、登录尝试额度改为 Redis TTL 状态；验证码和登录额度消费使用 Lua 原子校验/删除或递减，避免并发重复消费
  - HTTP metrics 改为 Redis 累计，多副本告警读取统一快照；指标写入失败只记录日志，不改变已经完成的 HTTP 响应
  - Redis I/O 全部移出 `store.update()`；登录成功后若清理 Redis 尝试令牌失败，会补偿删除刚创建的 session，避免接口返回 503 却残留有效会话
  - Redis 不可用时认证链路明确返回 `503 RUNTIME_STATE_UNAVAILABLE`；应用关闭时主动关闭 RuntimeState 客户端
- **Review 修复**：
  - 无登录尝试令牌时不再发送无意义的过期 cookie，恢复既有登录响应兼容性
  - 补 Redis 断线后同端口恢复及客户端自动重连测试
  - 补 Redis `DEL` 故障注入测试，确认登录失败时 JSON store 不残留新会话
  - Redis 测试替身补 socket `error` 监听，避免客户端主动断开导致测试进程出现未处理 `ECONNRESET`
- **验证**：✅ shared/API typecheck 与 build；✅ `tests/runtime-state.test.mjs`（7/7）；✅ `tests/auth-flow.test.mjs`（9/9）；✅ `tests/api-flow.test.mjs`（12/12）；✅ `tests/api-route-modules.test.mjs` + `tests/web-routes.test.mjs`（24/24）；✅ 定向 ESLint / Prettier；✅ `npm audit`（0 vulnerabilities）
