# Imagora 生产部署与验证手册

> 用途:本地/服务器用 Docker Compose 部署生产形态(Postgres + Redis + API + Worker + Web),并验证第 0 档上线阻塞项是否修复。
> 适用:`infra/docker-compose.prod.yml` 一套五服务 + 一个一次性 `migrate` 建表任务。
> 最后更新:2026-07-15

---

## 0. 这套流程在干嘛(先看懂)

一条命令 `npm run deploy:local-prod` 会:
1. 检查 `.env.production` 是否存在(缺则中文报错中止)
2. 用 `.env.production` 做变量替换
3. `docker compose ... up -d --build` —— 构建镜像 + 后台起容器

起来的容器:

| 服务 | 作用 | 依赖 |
|---|---|---|
| `postgres` | 数据库 | —— |
| `redis` | 队列 / 限流 | —— |
| `migrate` | **一次性**跑 `prisma migrate deploy` 建表,跑完即退出 | postgres 健康 |
| `api` | REST API | migrate 成功完成 + postgres/redis 健康 |
| `worker` | 图片生成消费 | migrate 成功完成 + api 健康 |
| `web` | 前端 | api 健康 |

> `migrate` 是第 0 档新增的关键服务:空库首次部署靠它自动建表,api/worker 用 `service_completed_successfully` 等它跑完再启动。

## 1. 前提条件(两个,缺一不可)

### 1.1 Docker Desktop 已启动

- 开始菜单搜 `Docker Desktop` 打开,等托盘鲸鱼图标变**稳定绿色**(首次启动 1-3 分钟)。
- 验证 daemon 真起来了:

```powershell
docker version --format '{{.Server.Version}}'
```

  返回一个版本号(如 `29.6.1`)且退出码 0 = OK。
  若报 `failed to connect to the docker API at npipe:...` = daemon 还没起,继续等。

### 1.2 `.env.production` 已就绪

```powershell
# 从模板复制(保留模板,别重命名)
Copy-Item .env.production.example .env.production
```

- **本轮只验"起容器+建表"**:只需把 `POSTGRES_PASSWORD` 从 `replace-password` 改成一个真密码(如 `imagora123456`),其余占位值可暂不动。
  - 原因:`replace-` 开头的占位密码会被 postgres 校验挡下;数据库三件套 `POSTGRES_USER/PASSWORD/DB` 是自带 postgres 容器要用的。
  - `WEB_ORIGIN` 保持模板的 `https://imagora.example.com`,**别改成 localhost**(生产配置校验会拒 localhost)。
- **要验真实业务**(真出图/真收钱/真存图):需填真实凭据 `OPENAI_*` / `S3_*` / `STRIPE_*` / `SMTP_*`,门槛高、第一次不建议。

> `.env.production` 已在 `.gitignore` 内,填了真值不会被提交。改的是 `.env.production`,别改 `.env.production.example`(模板)。

## 2. 部署命令(第 3 步)

在项目根目录 `e:\个人项目\Imagora`:

```powershell
npm run deploy:local-prod
```

- **首次很慢**(下基础镜像 + `npm ci` 装 400+ 包 + 编译),可能 5-20 分钟。
- 屏幕滚一大堆构建日志属正常。
- 手动等价命令(脚本内部实际执行的):

```powershell
docker compose --env-file .env.production -f infra/docker-compose.prod.yml up -d --build
```

## 3. 验证命令(第 4 步)

### 3.1 看容器状态

```powershell
docker compose -f infra/docker-compose.prod.yml --env-file .env.production ps
```

**成功**:`postgres`/`redis`/`api`/`worker`/`web` 在列,`api` STATUS 为 `healthy` 或 `running`;`migrate` 已 `Exited (0)`。

### 3.2 【核心】看 migrate 建表(验第 0 档任务 3)

```powershell
docker compose -f infra/docker-compose.prod.yml --env-file .env.production logs migrate
```

**成功**:日志出现 `Applying migration ...` / `migrations ... applied` / `The following migration(s) have been applied`。
→ 证明空库自动建表成功。

### 3.3 看 API 有没有配置自杀(验第 0 档任务 2)

```powershell
docker compose -f infra/docker-compose.prod.yml --env-file .env.production logs api
```

**成功**:看到 API 监听端口日志,**没有** `Unsafe production config: ...` 抛错退出。
→ 证明告警变量已正确透传进容器。

### 3.4 健康检查

```powershell
curl http://127.0.0.1:4100/health
```

**成功**:返回 JSON(不是连接被拒)。

## 4. 清理(第 5 步,可选)

```powershell
# 停容器,保留数据卷
docker compose -f infra/docker-compose.prod.yml --env-file .env.production down

# 连数据库数据一起删(慎用)
docker compose -f infra/docker-compose.prod.yml --env-file .env.production down -v
```

第一次验证建议**不加 `-v`**,保留数据。

## 5. 已知问题与解决

### 问题 1:`npm ci` 网络被重置 `ECONNRESET` / `Error: aborted`(2026-07-15 实测踩到)

**现象**:构建跑到 `RUN npm ci` 阶段,下载十几分钟后报:

```
npm error Error: aborted
npm error   code: 'ECONNRESET',
npm error   retriesLeft: 2
...
ERROR: process "/bin/sh -c npm ci" did not complete successfully: exit code: 1
```

其余镜像的 `npm ci` 被 `CANCELED`。

**根因**:容器内 `npm ci` 从官方源 `registry.npmjs.org` 下 400+ 包,国内直连极易被掐断。**与代码无关,是构建环境网络问题**。

**注意**:构建能进到 `npm ci` 这一步,已证明第 0 档改动全部生效(env 检查通过、变量替换成功、四镜像正常进入装依赖)。卡的是后续网络环节。

**解决方案(择一,待定,尚未应用)**:

- **方案 A · Dockerfile 内配国内 npm 镜像源(改动最小,推荐国内环境)**
  在四个 `infra/Dockerfile.*` 的 `RUN npm ci` 前加一行:
  ```dockerfile
  RUN npm config set registry https://registry.npmmirror.com
  RUN npm ci
  ```
  - 优点:一劳永逸,任何机器构建都走国内源。
  - 缺点:改了 4 个 Dockerfile;镜像源写死(可用 ARG 参数化)。

- **方案 B · 用 build arg 传镜像源(不写死,更干净)**
  Dockerfile 加 `ARG NPM_REGISTRY=https://registry.npmjs.org`,`RUN npm config set registry $NPM_REGISTRY && npm ci`;compose build args 传国内源。
  - 优点:默认官方源,国内构建时才切,不污染 Dockerfile 语义。

- **方案 C · 配 Docker 代理 / 挂梯子**
  不改代码,给 Docker Desktop 配 HTTP 代理(Settings → Resources → Proxies),让容器构建走代理出网。
  - 优点:零代码改动。
  - 缺点:依赖本机代理环境,换机器要重配。

> ⚠️ 改 Dockerfile 属构建行为变更,应用前需确认。方案未定前不动。

## 6. 排障速查

| 现象 | 可能原因 | 处理 |
|---|---|---|
| `failed to connect to the docker API at npipe:...` | Docker daemon 没起 | 开 Docker Desktop,等鲸鱼变绿 |
| 脚本报"缺少环境文件 .env.production" | 没复制模板 | `Copy-Item .env.production.example .env.production` |
| `required variable POSTGRES_USER is missing` | 模板没填 / 没加 `--env-file` | 检查 `.env.production` 三件套;脚本已内置 `--env-file`,手动跑记得带 |
| `npm ci` `ECONNRESET` | 网络掐断(见问题 1) | 配国内源 / 代理 |
| `api` 日志 `Unsafe production config: xxx` | 某生产必填变量缺失/非法 | 按报错的变量名补 `.env.production` |
| `migrate` 容器 `Exited (1)` | 数据库连不上 / 迁移报错 | 看 `logs migrate` + 确认 postgres 健康 |
| `api`/`worker` 一直不起 | 在等 `migrate` 完成 | 先修好 `migrate`,依赖链会自动放行 |
