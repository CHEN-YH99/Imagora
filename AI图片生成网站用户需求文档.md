## 2. 项目背景

随着 AI 图片生成能力成熟，个人创作者、设计师、电商运营、自媒体团队和企业内容团队对低成本、高效率生成图片素材的需求持续增长。本项目计划建设一个 AI 图片生成网站，用户可通过自然语言提示词、参数配置和参考图上传，快速生成、管理、下载和复用图片资产。

本需求文档重点描述用户视角的目标、场景、功能范围和验收标准，不限定具体模型供应商。模型接入方案、成本策略、支付通道和合规策略需在技术方案阶段进一步确认。

## 3. 产品目标

### 3.1 核心目标

1. 用户能够通过输入提示词快速生成高质量图片。
2. 用户能够方便地调整图片风格、比例、尺寸、数量、质量等生成参数。
3. 用户能够查看生成历史，并对图片进行收藏、下载、再次生成或二次编辑。
4. 平台能够对用户、任务、积分、订单、内容安全和资源成本进行基础管理。
5. 管理员能够运营用户、套餐、生成记录、敏感内容和系统配置。

### 3.2 成功指标

| 指标 | 说明 |
| --- | --- |
| 首次生成完成率 | 新用户完成第一次图片生成的比例 |
| 生成成功率 | AI 任务成功返回图片的比例 |
| 平均生成耗时 | 从提交任务到图片可查看的平均时间 |
| 图片下载率 | 生成图片被用户下载的比例 |
| 付费转化率 | 注册用户购买积分或套餐的比例 |
| 次日留存率 | 用户第二天再次访问或生成图片的比例 |

## 4. 用户角色

| 用户角色 | 典型需求 | 关键关注点 |
| --- | --- | --- |
| 游客 | 了解产品能力，查看示例，尝试有限次数生成 | 低门槛、无需复杂配置 |
| 注册用户 | 生成图片、管理历史、下载作品 | 生成效果、速度、稳定性 |
| 付费用户 | 使用更多额度、更高质量、更高并发能力 | 成本清晰、权益稳定 |

| 管理员 | 管理用户、套餐、内容、任务和系统配置 | 可控、可追溯、可运营 |

## 5. 需求假设

1. 产品初期以 Web 端为主，优先适配桌面端，同时保证移动端可正常使用。
2. 用户需要登录后才能保存历史记录、下载高清图片和使用付费额度。
3. 图片生成采用异步任务机制，避免用户长时间卡在请求中。
4. 图片生成会产生模型调用、存储和带宽成本，因此需要积分或额度体系。
5. 具体 AI 模型、支付渠道、对象存储、内容审核供应商暂未指定，本文只定义业务需求。

## 7. 用户核心流程

### 7.1 游客体验流程

1. 用户进入首页。
2. 查看平台示例图片和基础能力。
3. 输入提示词或选择示例提示词。
4. 系统提示登录或允许有限次数试用。
5. 用户完成登录后提交生成任务。
6. 系统展示生成进度和结果。
7. 用户可下载低清图或引导升级获取高清图。

### 7.2 注册用户生成图片流程

1. 用户登录平台。
2. 进入图片生成页面。
3. 输入提示词。
4. 选择风格、比例、数量、质量等参数。
5. 查看本次预计消耗积分。
6. 提交生成任务。
7. 系统展示任务状态：排队中、生成中、成功、失败。
8. 图片生成成功后展示结果。
9. 用户可下载、收藏、复制提示词、再次生成或删除记录。

### 7.3 付费流程

1. 用户进入套餐或充值页面。
2. 查看不同套餐权益、价格和积分数量。
3. 选择套餐并提交订单。
4. 跳转支付。
5. 支付成功后系统增加用户积分或开通会员权益。
6. 用户可在账户页面查看订单记录和权益状态。

### 7.4 管理员运营流程

1. 管理员登录后台。
2. 查看用户增长、生成任务、消耗成本和收入概览。
3. 查询用户生成记录。
4. 对违规内容进行处理。
5. 配置套餐、积分规则、模型参数和敏感词策略。
6. 查看失败任务与系统告警。

## 11. 非功能需求

### 11.1 性能需求

1. 首页首屏加载应尽量控制在 3 秒内。
2. 图片生成任务提交接口应在 2 秒内返回任务 ID。
3. 任务状态轮询或推送应能及时反馈状态变化。
4. 历史记录列表应支持分页，避免一次性加载大量图片。
5. 图片资源应使用缩略图和懒加载，减少带宽浪费。

### 11.2 稳定性需求

1. AI 模型调用失败时，系统应记录错误并给出用户可理解提示。
2. 支付回调必须支持幂等处理，避免重复发放积分。
3. 积分扣减和任务创建必须保持一致，不能出现扣了分但无任务记录的情况。
4. 图片上传和下载应有异常处理。
5. 管理后台关键操作应记录操作日志。

### 11.3 安全需求

1. 用户密码必须加密存储。
2. 用户只能访问自己的生成记录和图片。
3. 管理后台必须进行权限校验。
4. 上传文件必须限制格式、大小和内容。
5. 支付回调必须校验签名。
6. 防止常见 Web 安全风险，包括 XSS、CSRF、SQL 注入、越权访问。

### 11.4 合规需求

1. 用户提交内容和生成结果需经过内容安全策略。
2. 平台需明确用户生成内容的使用条款和责任边界。
3. 如支持商用，需明确商用授权范围。
4. 平台需提供隐私政策和用户协议。
5. 用户数据删除、账号注销和数据导出需求需在合规阶段确认。

### 11.5 可用性需求

1. 生成表单应保持简洁，基础用户无需理解复杂模型参数。
2. 高级参数可折叠，避免干扰普通用户。
3. 错误信息应说明原因和下一步操作。
4. 积分消耗必须在提交前明确展示，别让用户生成完才发现被扣，太阴间。
5. 移动端应能完成注册、生成、预览和下载等核心操作。

## 13. 业务规则

1. 未登录用户最多只能体验有限次数，具体次数待运营确认。
2. 用户提交生成任务前必须有足够积分。
3. 任务创建成功后扣减积分。
4. 因系统或模型异常导致失败时，应自动退还积分。
5. 因用户输入违规内容被拦截时，不应扣减积分。
6. 用户删除历史记录不代表物理删除图片文件，实际清理策略需由存储策略决定。
7. 管理员删除违规内容后，用户前台不可继续访问。
8. 套餐价格、积分数量、有效期和权益变更不应影响已支付订单的既有权益，除非用户协议另有说明。
9. 支付成功但权益未到账时，系统需支持订单补偿或人工处理。
10. 生成图片的公开展示必须获得用户授权。

## 14. 验收标准

### 14.1 MVP 验收标准

1. 用户可以完整完成注册、登录、生成图片、查看结果、下载图片的闭环。
2. 用户可以查看自己的生成历史和积分余额。
3. 图片生成任务支持成功、失败、排队中、生成中等状态。
4. 余额不足、提示词违规、模型失败等异常场景有明确提示。
5. 管理员可以查看用户、任务、图片和订单基础数据。
6. 积分扣减、退还和流水记录准确。
7. 移动端和桌面端核心流程可用。
8. 基础安全校验、权限校验和内容拦截生效。

### 14.2 不通过条件

1. 用户可查看或下载他人的私有图片。
2. 支付成功后权益经常不到账且无补偿机制。
3. 任务失败仍扣积分且无退还记录。
4. 违规提示词或图片可绕过安全检查。
5. 页面核心流程在移动端无法使用。
6. 管理后台可被普通用户访问。
7. 图片生成任务状态丢失，用户无法确认结果。

## 17. 风险与待确认事项

| 类型 | 待确认问题 | 影响 |
| --- | --- | --- |
| 模型供应商 | 使用哪家 AI 图片生成模型 | 影响成本、质量、速度和合规 |

## 20. 结论

AI 图片生成网站的首要目标不是堆一堆“高级参数”吓人，而是让用户在最短路径内完成从想法到可用图片的闭环。MVP 应优先保证生成入口、任务状态、图片管理、积分体系、支付闭环和后台监管能力稳定可用。模型能力、模板市场、图生图和局部重绘等增强功能可以在核心闭环验证后逐步扩展。
